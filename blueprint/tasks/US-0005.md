# US-0005

## Title

Create and Remove Knowledge Object Relations

## Goal

Introduce directed relations between Knowledge Objects.

A relation connects one Knowledge Object to another using an explicit relation type.

This task implements relation creation and removal in memory.

Graph traversal and persistence adapters are out of scope.

---

## Domain Model

Create a new immutable domain entity or value object representing a relation.

Suggested location:

`backend/core/domain/entities/knowledge_object_relation.py`

The relation shall contain:

- relation_id
- organization_id
- source_object_id
- target_object_id
- relation_type
- created_at

Use domain Value Objects where appropriate.

---

## Relation Direction

Relations are directed.

Example:

```text
Instruction --addresses--> Risk
```

The inverse relation must not be created automatically.

---

## Relation Type

Create:

`KnowledgeObjectRelationType`

Suggested location:

`backend/core/domain/value_objects/knowledge_object_relation_type.py`

Rules:

- non-empty string;
- normalized to lowercase;
- leading and trailing whitespace removed;
- immutable.

Examples:

- `addresses`
- `requires`
- `belongs_to`
- `supersedes`

Do not create a fixed enum.

The platform must support extensible relation types.

---

## Domain Rules

A relation cannot connect an object to itself.

The source and target objects must:

- exist;
- belong to the same organization;
- not have status `DELETED`.

Creating the same directed relation twice shall raise:

`DuplicateKnowledgeObjectRelation`

A relation is considered duplicate when these fields match:

- organization_id
- source_object_id
- target_object_id
- relation_type

The reverse relation is not a duplicate.

Example:

```text
A --uses--> B
```

is different from:

```text
B --uses--> A
```

Removing a relation that does not exist shall raise:

`KnowledgeObjectRelationNotFound`

---

## Domain Exceptions

Add:

- `KnowledgeObjectRelationNotFound`
- `DuplicateKnowledgeObjectRelation`
- `SelfReferencingKnowledgeObjectRelation`
- `CrossOrganizationKnowledgeObjectRelation`

All exceptions shall inherit from:

`SafetyMainDomainError`

Exceptions must preserve relevant identifiers and provide readable messages.

---

## Domain Events

Create:

- `KnowledgeObjectRelationCreated`
- `KnowledgeObjectRelationRemoved`

Events shall inherit from:

`BaseDomainEvent`

Do not publish or process events.

---

## Repository Contract

Create:

`KnowledgeObjectRelationRepositoryContract`

Suggested operations:

```python
add(relation) -> None
get(relation_id) -> KnowledgeObjectRelation
remove(relation_id) -> None
exists(
    organization_id,
    source_object_id,
    target_object_id,
    relation_type,
) -> bool
```

Do not add relation methods to `KnowledgeObjectRepositoryContract`.

Knowledge Objects and Relations must have separate repository contracts.

---

## In-Memory Adapter

Implement an in-memory relation repository.

It shall:

- store relations by relation ID;
- detect duplicate relations;
- return typed domain exceptions;
- preserve immutable relation objects;
- contain no business workflow logic.

---

## Unit of Work

Expose the relation repository through `UnitOfWorkContract`.

Suggested property:

```python
relations: KnowledgeObjectRelationRepositoryContract
```

Update the in-memory Unit of Work accordingly.

Rollback must restore both:

- Knowledge Objects;
- Relations.

---

## Application Layer

Implement:

- CreateKnowledgeObjectRelationCommand
- RemoveKnowledgeObjectRelationCommand
- CreateKnowledgeObjectRelationHandler
- RemoveKnowledgeObjectRelationHandler

The create handler shall:

1. load source Knowledge Object;
2. load target Knowledge Object;
3. validate organization and lifecycle rules;
4. create the relation;
5. persist it;
6. commit once.

The remove handler shall:

1. load the relation;
2. remove it;
3. commit once.

Do not commit inside repositories or domain services.

---

## Domain Service

Create:

`KnowledgeObjectRelationService`

It shall contain relation-specific domain validation and relation creation behavior.

Do not place relation validation inside the application handlers when it represents a domain rule.

---

## Tests

Add unit and integration tests covering:

- valid directed relation creation;
- relation type normalization;
- relation immutability;
- duplicate relation rejection;
- self-reference rejection;
- cross-organization rejection;
- source object not found;
- target object not found;
- deleted source rejection;
- deleted target rejection;
- valid relation removal;
- missing relation removal;
- reverse relation allowed;
- Unit of Work rollback restores relation state.

Verify that existing Knowledge Object tests continue to pass.

---

## Acceptance Criteria

- Directed Knowledge Object Relations implemented.
- Knowledge Object and Relation repositories remain separate.
- Relation types are extensible and not hard-coded as an enum.
- Duplicate, self-reference and cross-organization relations are rejected.
- Deleted objects cannot participate in new relations.
- Domain Events are created but not published.
- Unit of Work manages both repositories.
- Rollback restores both Knowledge Objects and Relations.
- All tests pass.
- Ruff passes.
- Import validation passes.
- Architecture tests pass.

---

## Non-Goals

- Graph traversal.
- Relation search.
- Automatic inverse relations.
- Relation metadata.
- Relation version history.
- PostgreSQL.
- SQLAlchemy.
- API endpoints.
- Event publishing.
